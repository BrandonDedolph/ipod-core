# 01 — PortalPlayer PP5022 SoC

The iPod Video 5G/5.5G runs on the PP5022 SoC: dual ARM7TDMI cores
("CPU" and "COP" — co-processor), tightly-coupled IRAM, integrated
peripheral block, and shared external SDRAM. Despite the marketing
name, the chip behaves as a member of the PP502x family; most of the
Rockbox platform code targets PP5020/5022/5024 uniformly via
`firmware/target/arm/pp/`.

## Memory map

External SDRAM natively lives at `0x10000000`; the *loaded firmware
itself* remaps it to `0x00000000` early in its crt0 so exception
vectors live at low addresses. The chainloading bootloader
(ipodloader2) restores the Apple-ROM MMAP state before jumping to the
image at `0x10000000` — it does NOT hand off a 0x0 mapping (see
08-boot-dock.md, "Boot ROM → our code: handoff state").

| Region        | 5G (32 MB)             | 5.5G (64 MB)           | Notes |
|---------------|------------------------|------------------------|-------|
| SDRAM         | `0x10000000`–`0x11FFFFFF` | `0x10000000`–`0x13FFFFFF` | Remapped to `0x00000000` after boot |
| ROM / flash   | `0x20000000`–`0x200FFFFF` | same                  | 1 MB; firmware partition lives below |
| MMIO          | `0x60000000`–`0x700FFFFF` | same                  | Peripherals + SoC config |
| IRAM          | `0x40000000`–`0x4001FFFF` | same                  | Tightly-coupled, ~96 KB usable to user code |
| MMAP regs     | `0xF000F000`–`0xF000F03F` | same                  | Logical→physical remap registers |

**RAM size detection** (`firmware/target/arm/pp/crt0-pp.S` lines
144–160): the bootloader writes test patterns to `0x01FFFFFF` and
`0x03FFFFFF` and reads them back to detect 32 MB vs 64 MB. The result
is preserved in `probed_ramsize` for later use (e.g., default battery
capacity differs between 5G and 5.5G).

**Memory remap (MMAP0):** `MMAP0_LOGICAL` (`0xF000F000`) is set to
`0x00003C00` — logical base `0x00000000` OR'd with the window-size
mask, which lives in the *logical* register: `0x3C00` selects a 64 MB
window, `0x3E00` a 32 MB one. Rockbox uses the 64 MB mask for every
iPod Video build (RAM size is probed at runtime, not encoded in MMAP),
so one image covers 5G and 5.5G. `MMAP0_PHYSICAL` (`0xF000F004`) is
set to `0x10000000` with flags `0x0F84`: `0x0F00` = read | write |
data | code permission bits (Rockbox `pp5020.h` `MMAP_PHYS_*_MASK`);
`0x84` is undocumented but present in every known user. (`0x3F84` is
the PP5002 flags value — ipodloader2 also uses it on PP502x and it
works; we follow Rockbox's PP502x choice.) Write order: logical first,
then physical; no barrier needed. Done in IRAM so the remapping code
isn't pulled out from under itself.

*(Corrected 2026-06-10 against Rockbox `crt0-pp.S` + `pp5020.h`: an
earlier revision gave `MMAP0_LOGICAL = 0x00000000`, omitting the mask
bits — a value no known firmware writes.)*

## Clock tree

The PP5022 derives all clocks from a 24 MHz crystal through a
configurable PLL. Four canonical operating points:

| Name             | Freq      | Use                          |
|------------------|-----------|------------------------------|
| `CPUFREQ_SLEEP`  | 32 768 Hz | Deep sleep (PLL off, 32 kHz oscillator) |
| `CPUFREQ_DEFAULT`| 24 MHz    | Crystal direct, PLL bypassed |
| `CPUFREQ_NORMAL` | 30 MHz    | UI / unboosted execution     |
| `CPUFREQ_MAX`    | 80 MHz    | Boosted; codecs, decode-heavy work |

(Defined in `firmware/target/arm/pp/system-target.h`.)

**Key registers** (`firmware/export/pp5020.h`):

| Register       | Address      | Purpose |
|----------------|--------------|---------|
| `PLL_CONTROL`  | `0x60006034` | PLL enable, multiplier, post-divider |
| `PLL_STATUS`   | `0x6000603C` | Lock status (bit 31) |
| `CLOCK_SOURCE` | `0x60006020` | 4-bit nibble per clock domain (see below) |
| `DEV_INIT2`    | `0x70000020` | PLL power gate (bit 30) |
| `DEV_TIMING1`  | `0x70000034` | Memory / peripheral timing for current clock |

**`CLOCK_SOURCE` source codes** (per nibble):

| Code | Source       |
|------|--------------|
| 0    | 32 kHz       |
| 1    | 16 MHz       |
| 2    | 24 MHz crystal |
| 3    | 33 MHz       |
| 4    | 48 MHz       |
| 5    | SLOW divider |
| 6    | FAST divider |
| 7    | PLL output   |

**Frequency switch sequence** (`firmware/target/arm/pp/system-pp502x.c`
lines 379–475). Boosting from 30 MHz to 80 MHz on PP5022 — the actual
order matters because the other core sees clocks change while it's
still executing:

1. `DEV_INIT2 |= 0x40000000` — power up PLL.
2. `PLL_CONTROL |= 0x88000000` — enable PLL.
3. **Suspend the other core** via `scale_suspend_core(true)`. From this
   point until step 9 the other core is parked (see *Dual core* below).
4. `CLOCK_SOURCE = 0x20002222` — switch all domains to 24 MHz crystal
   (intermediate, no PLL dependency — keeps the bus running while we
   reconfigure the PLL).
5. `DEV_TIMING1 = 0x00000303` — relax timing for the slow clock.
6. `PLL_CONTROL = 0x8A020A03` — pre-stage write (10/3 multiplier; not
   used directly, but the controller wants this intermediate write
   before the final value).
7. `PLL_CONTROL = 0x8A121403` — program PLL for 80 MHz (final value:
   multiplier = 20/3, ÷2 post-divider, on a 24 MHz reference).
8. Spin on `PLL_STATUS & 0x80000000` until lock.
9. `CLOCK_SOURCE = 0x20007777` — route all domains to PLL output.
10. `DEV_TIMING1 = 0x00000808` — high-speed timing.
11. **Resume the other core** via `scale_suspend_core(false)`.

For 30 MHz: same shape with `PLL_CONTROL = 0x8A220501` (multiplier
5/1, ÷4 post-divider).

For sleep (32 kHz): disable post-dividers (`PLL_CONTROL |= 0x0C000000`),
switch to `CLOCK_SOURCE = 0x20000000`, disable PLL, then drop power
(`DEV_INIT2 &= ~0x40000000`).

**Boost discipline:** the firmware tracks a boost counter; UI work
runs unboosted at 30 MHz; codecs request a boost during decode. The
locking mechanism (`set_cpu_frequency__lock` /
`set_cpu_frequency__unlock`) uses a corelock to ensure the two cores
agree on the current frequency atomically.

**PP5022 has NO pre-stage write.** The 11-step sequence above lists a
pre-stage `PLL_CONTROL = 0x8A020A03` (step 6) before the final value.
That intermediate write — together with the `PLL_STATUS = 0xD19B`
">66 MHz unlock" poke and the doubled setup write — belongs to the
**PP5020** code path only (`system-pp502x.c` lines 401–405). On
**PP5022/PP5024** (our silicon) the PLL is programmed with a **single**
`PLL_CONTROL` write and then the lock bit is polled: `0x8A121403` for
80 MHz (line 407) and `0x8A220501` for 30 MHz (line 449). There is no
pre-stage and no unlock poke on PP5022 for either target. The
driver sequences below reflect the PP5022 single-write reality.

### For the driver — minimal single-core boost/unboost sequence

Our driver is single-core: the COP is parked asleep via `COP_CTL`
during crt0 (`boot/crt0.S`) and is never woken in Phase 1. The
`scale_suspend_core(true/false)` bracket that Rockbox wraps around the
`CLOCK_SOURCE`/`DEV_TIMING1` writes (`system-pp502x.c` lines 329–347)
exists only to stop a *running* second core from executing across the
bus-source switch — and its cross-core PROC_CTL writes are themselves
compiled out (`IF_COP(...)`) on a single-core build. With the COP
already parked (not fetching from the bus), there is nothing to protect,
so we **omit** the suspend/resume entirely. None of the source-switch or
PLL-program writes depends on the other core being suspended for its own
correctness — suspension is purely about the *other* core's execution.
`// TODO(dual-core): reinstate scale_suspend_core once the COP is booted.`

The PLL power bit (`DEV_INIT2 |= 0x40000000`, bit 30) is **not** assumed
to be set at entry — the handoff state from ipodloader2 is not
guaranteed, and Rockbox itself sets it explicitly on every PLL path
(`system-pp502x.c` lines 390, 434) rather than relying on prior state.
`clock_init` sets it itself.

**Bounded lock poll — mandatory.** The `PLL_STATUS & 0x80000000` lock
wait MUST be a **bounded** spin (a fixed iteration/`USEC_TIMER` cap),
never an unbounded `while`. The clicky emulator never sets the lock bit,
and a dead PLL on real silicon never locks; an unbounded wait hangs boot
in both. Bounded-then-proceed is both emulator-safe and the correct
hardware degrade posture: if the PLL fails to lock we continue at the
last stable source rather than wedging.

**`clock_init` (24 MHz boot → 30 MHz, `CPUFREQ_NORMAL`).** Boot enters
at `CPUFREQ_DEFAULT` = 24 MHz (crystal direct, PLL bypassed). To reach
the UI operating point:

1. `DEV_INIT2 |= 0x40000000` — power up the PLL (bit 30). Set it
   ourselves; do not assume handoff state.
2. `PLL_CONTROL |= 0x88000000` — enable the PLL.
3. *(suspend omitted — single core; see note above)*
4. `CLOCK_SOURCE = 0x20002222` — route all domains to the 24 MHz crystal
   (PLL-independent intermediate; keeps the bus alive while we reprogram
   the PLL).
5. `DEV_TIMING1 = 0x00000303` — relax memory/peripheral timing for the
   slow clock.
6. *(resume omitted)*
7. `PLL_CONTROL = 0x8A220501` — program the PLL for 30 MHz
   ((5/1 × 24 MHz) ÷ 4). **Single write, no pre-stage** on PP5022.
8. **Bounded** spin on `PLL_STATUS & 0x80000000` until lock. On lock,
   continue; on timeout **degrade** — skip steps 10–11, stay on the
   24 MHz crystal (report `CPUFREQ_DEFAULT`), never routing the bus onto
   an unlocked PLL.
9. *(suspend omitted)*
10. `DEV_TIMING1 = 0x00000303` — timing for the 30 MHz operating point.
11. `CLOCK_SOURCE = 0x20007777` — route all domains to PLL output.
12. *(resume omitted)*

**`cpu_boost` (30 → 80 MHz, `CPUFREQ_MAX`).** Starting from the PLL-locked
30 MHz state:

1. `DEV_INIT2 |= 0x40000000` — idempotent (PLL already powered); mirrors
   Rockbox's unconditional set.
2. `PLL_CONTROL |= 0x88000000` — idempotent (PLL already enabled).
3. *(suspend omitted)*
4. `CLOCK_SOURCE = 0x20002222` — drop off the PLL to the 24 MHz crystal
   before reprogramming it.
5. `DEV_TIMING1 = 0x00000303` — relax timing.
6. *(resume omitted)*
7. `PLL_CONTROL = 0x8A121403` — program the PLL for 80 MHz
   ((20/3 × 24 MHz) ÷ 2). **Single write, no pre-stage** on PP5022.
8. **Bounded** spin on `PLL_STATUS & 0x80000000` until lock. On lock,
   continue; on timeout **degrade** — skip steps 10–11 and stay on the
   24 MHz crystal, never routing onto an unlocked PLL.
9. *(suspend omitted)*
10. `DEV_TIMING1 = 0x00000808` — high-speed timing.
11. `CLOCK_SOURCE = 0x20007777` — route all domains to PLL output.
12. *(resume omitted)*

**`cpu_unboost` (80 → 30 MHz, back to `CPUFREQ_NORMAL`).** Register-for-
register identical to `clock_init` (both are just "select
`CPUFREQ_NORMAL`"); only the entry frequency differs:

1. `DEV_INIT2 |= 0x40000000` — idempotent.
2. `PLL_CONTROL |= 0x88000000` — idempotent.
3. *(suspend omitted)*
4. `CLOCK_SOURCE = 0x20002222` — drop off the PLL to the 24 MHz crystal.
5. `DEV_TIMING1 = 0x00000303` — relax timing.
6. *(resume omitted)*
7. `PLL_CONTROL = 0x8A220501` — program the PLL for 30 MHz. **Single
   write, no pre-stage** on PP5022.
8. **Bounded** spin on `PLL_STATUS & 0x80000000` until lock. On lock,
   continue; on timeout **degrade** — skip steps 10–11 and stay on the
   24 MHz crystal, never routing onto an unlocked PLL.
9. *(suspend omitted)*
10. `DEV_TIMING1 = 0x00000303` — 30 MHz timing.
11. `CLOCK_SOURCE = 0x20007777` — route all domains to PLL output.
12. *(resume omitted)*

## Dual core: CPU and COP

Both cores are ARM7TDMI; they share IRAM and SDRAM, and have a small
mailbox peripheral for IPC. Cores are distinguished by reading
`PROCESSOR_ID` at `0x60000000`:

```c
#define PROC_ID_CPU  0x55
#define PROC_ID_COP  0xAA
```

Per-core control via `CPU_CTL` (`0x60007000`) and `COP_CTL`
(`0x60007004`); the macro `PROC_CTL(core)` selects between them.

**Sleep / wake:**

| Bit          | Value         | Meaning |
|--------------|---------------|---------|
| `PROC_SLEEP` | `0x80000000`  | Sleep until interrupt |
| `PROC_WAIT_CNT` | `0x40000000` | Sleep until countdown |
| `PROC_WAKE_INT` | `0x20000000` | Fire IRQ on wake |
| `PROC_CNT_USEC` | `0x02000000` | Counter ticks at 1 µs |
| `PROC_CNT_MSEC` | `0x01000000` | Counter ticks at 1 ms |
| `[7:0]`      | —             | Read: cycles remaining; write: cycles to skip |

**Always insert 3 NOPs after writing PROC_CTL** — the pipeline needs
a beat for the write to take effect.

**Mailbox IPC** (registers at `0x60001000`):

| Register       | Address      | Behavior |
|----------------|--------------|----------|
| `MBX_MSG_STAT` | `0x60001000` | Read mailbox bits |
| `MBX_MSG_SET`  | `0x60001004` | Atomic OR (set bits) |
| `MBX_MSG_CLR`  | `0x60001008` | Atomic AND-NOT (clear bits) |
| `CPU_QUEUE`    | `0x60001010` | CPU writes bit 29 to signal COP |
| `COP_QUEUE`    | `0x60001020` | COP writes bit 29 to signal CPU |

Wake protocol (`firmware/target/arm/pp/thread-pp.c` lines 319–346):

1. Set wake-intent flags via `MBX_MSG_SET`.
2. Spin until target core signals sleep-intent (`0x4 << core_id`).
3. If target's `PROC_CTL` shows `PROC_SLEEP`, write 0 to clear it.
4. Clear wake procedure flag via `MBX_MSG_CLR`.

**COP boot:** the bootloader (running on CPU) wakes the COP after
SDRAM remap (`crt0-pp.S` lines 193–214). The COP redoes the remap on
its side and sets up its own stacks. CPU then waits for the COP to
finish kernel init before either core can do meaningful work.

**Core suspension during clock change** (`scale_suspend_core`,
`system-pp502x.c` 330–348): the active core writes `0x40000000` to the
*other* core's `PROC_CTL` (signal-wait), then writes `0x48000003` to
its own (stall 3 cycles); on resume, the active core stalls 31 cycles
and clears the other's control word. Interrupts are disabled across
the whole sequence.

## Interrupt controller

Two priority levels (low and high) per core, with separate enable /
disable / status registers.

**Low-priority** (`firmware/export/pp5020.h` lines 66–137):

| Register          | Address      | Purpose |
|-------------------|--------------|---------|
| `CPU_INT_STAT`    | `0x60004000` | CPU IRQ status |
| `COP_INT_STAT`    | `0x60004004` | COP IRQ status |
| `CPU_FIQ_STAT`    | `0x60004008` | CPU FIQ status |
| `COP_FIQ_STAT`    | `0x6000400C` | COP FIQ status |
| `INT_STAT`        | `0x60004010` | Shared status (logical OR) |
| `CPU_INT_EN`      | `0x60004024` | Write 1 to enable a bit |
| `CPU_INT_DIS`     | `0x60004028` | Write 1 to disable a bit |
| `CPU_INT_PRIORITY`| `0x6000402C` | Per-IRQ priority |
| `COP_INT_EN`      | `0x60004034` | (analogous for COP) |
| `COP_INT_DIS`     | `0x60004038` | |

**High-priority** (offset `+0x100`):

| Register         | Address      |
|------------------|--------------|
| `CPU_HI_INT_STAT`| `0x60004100` |
| `CPU_HI_INT_EN`  | `0x60004124` |
| `CPU_HI_INT_DIS` | `0x60004128` |

**IRQ numbers** of interest. The IRQ number is the absolute index;
sources at `# < 32` route through `CPU_INT_EN` / `CPU_INT_DIS` directly
(bit `# = N`); sources at `# ≥ 32` route through `CPU_HI_INT_EN` /
`CPU_HI_INT_DIS` at bit `(N − 32)`.

| #  | Source                  | Bank | Register bit |
|----|-------------------------|------|---------------|
| 0  | `TIMER1_IRQ`            | low  | `1 << 0`      |
| 1  | `TIMER2_IRQ`            | low  | `1 << 1`      |
| 4  | `MAILBOX_IRQ`           | low  | `1 << 4`      |
| 10 | `IIS_IRQ` (I²S audio)   | low  | `1 << 10`     |
| 20 | `USB_IRQ`               | low  | `1 << 20`     |
| 23 | `IDE_IRQ` (ATA)         | low  | `1 << 23`     |
| 25 | `FIREWIRE_IRQ`          | low  | `1 << 25`     |
| 26 | `DMA_IRQ`               | low  | `1 << 26`     |
| 30 | `HI_IRQ` (chained high) | low  | `1 << 30`     |
| 32 | `GPIO0_IRQ` (ports A–D) | high | `1 << 0`      |
| 33 | `GPIO1_IRQ` (ports E–H) | high | `1 << 1`      |
| 34 | `GPIO2_IRQ` (ports I–L) | high | `1 << 2`      |
| 36 | `SER0_IRQ` (UART 0)     | high | `1 << 4`      |
| 40 | `I2C_IRQ`               | high | `1 << 8`      |

So enabling `I2C_IRQ` from C looks like:

```c
CPU_HI_INT_EN = (1u << (40 - 32));   // == 1 << 8 == 0x100
```

This indirection is easy to get wrong; Rockbox's source uses
`#define I2C_MASK (1 << (I2C_IRQ - 32))` and similar macros to keep
each driver from re-deriving it.

**Vector remap:** `CACHE_CTL` bit `0x10` (`CACHE_CTL_VECT_REMAP`)
diverts the standard ARM exception vectors to `0x6000F100`; entries
are `INT_VECTOR_ENTRY(v)` = `*((vu32*)0x6000F100 + v)`.

## Timers and the system tick

The PP5022 exposes two general-purpose down-counters plus a free-running
microsecond counter in a small register block based at `0x60005000`.
Both counters are clocked at a fixed 1 MHz, so one count is one
microsecond regardless of the CPU/PLL clock state (`firmware/export/pp5020.h`
line 1161: `TIMER_FREQ` = 1 000 000). This is what makes them usable as a
stable time base while the core frequency scales.

**Register block** (`firmware/export/pp5020.h` lines 346–351):

| Register     | Address      | Meaning |
|--------------|--------------|---------|
| `TIMER1_CFG` | `0x60005000` | Timer 1 config: enable, IRQ-enable, reload period |
| `TIMER1_VAL` | `0x60005004` | Timer 1 live counter; reading it also acknowledges its IRQ |
| `TIMER2_CFG` | `0x60005008` | Timer 2 config (same layout as `TIMER1_CFG`) |
| `TIMER2_VAL` | `0x6000500C` | Timer 2 live counter |
| `USEC_TIMER` | `0x60005010` | Free-running 32-bit microsecond counter (read-only time source) |

`USEC_TIMER` is a monotonically-incrementing 1 MHz counter with no
configuration of its own — it is read directly for microsecond
timestamps and busy-wait delays. `TIMER1`/`TIMER2` are the programmable
periodic sources.

**`TIMERx_CFG` bit fields.** The value Rockbox programs into the config
register is `0xC0000000 | (period − 1)` (`firmware/target/arm/pp/kernel-pp.c`
line 48; the same shape appears for `TIMER2` in
`firmware/target/arm/pp/timer-pp.c` lines 32 and 61). Decoding that
constant against how the register is used:

| Bits    | Role |
|---------|------|
| `31`    | Timer enable (`0x80000000`). Clear to stop; `TIMER1_CFG = 0` disarms it. |
| `30`    | IRQ / periodic-reload enable (`0x40000000`). The counter reloads from the period field and re-fires each time it reaches zero. |
| `[26:0]`| Reload period, in microseconds, encoded as `period − 1`. A full period is `N` counts when this field holds `N − 1`. |

So `TIMERx_CFG` is armed as a self-reloading periodic timer in a single
write: the top two bits turn it on and make it recur, and the low field
sets the interval. There is no separate "start" strobe.

**The ~100 Hz system tick.** Our kernel drives its scheduler tick at
HZ = 100 (`firmware/kernel/include/tick.h` line 27: `HZ` = 100), i.e. a
10 ms period. In microseconds that is 1 000 000 / 100 = 10 000 µs, and
because the reload field counts microseconds directly the period value
written is 10 000 − 1 = 9 999 (`0x0000270F`). The full config word for a
100 Hz tick on TIMER1 is therefore:

```
TIMER1_CFG = 0xC0000000 | (10000 - 1);   // = 0xC000270F
```

Rockbox reaches the same value indirectly — its generic layer calls the
target `tick_start(1000 / HZ)` with the interval in *milliseconds* (10 ms),
and the target routine multiplies back to microseconds
(`interval_in_ms * 1000 − 1`) before OR-ing in `0xC0000000`
(`firmware/target/arm/pp/kernel-pp.c` lines 42–49). Either way the low
field ends up at 9 999.

The tick timer's interrupt is **`TIMER1_IRQ`, IRQ #0** — low bank, bit
`1 << 0` — as tabulated under *Interrupt controller* above; `TIMER2_IRQ`
is IRQ #1. After programming `TIMER1_CFG`, the source is unmasked by
enabling that bit in the CPU low-priority interrupt-enable register.

**For the driver.** The minimal sequence to start a periodic 100 Hz tick:

1. Disarm first: write `TIMER1_CFG = 0` (clears any stale enable/reload).
2. Read `TIMER1_VAL` once to clear a pending IRQ latched from a prior arm.
3. Arm periodic: `TIMER1_CFG = 0xC0000000 | (10000 - 1)` — enable +
   IRQ/reload + 10 ms period.
4. Unmask `TIMER1_IRQ` (IRQ #0) in the CPU interrupt-enable register.

Because bit 30 makes the hardware auto-reload, the ISR does **not** need
to reprogram the period each tick. The one action the ISR must take is to
**acknowledge** the interrupt: on this hardware the acknowledge is a read
of `TIMER1_VAL` (`firmware/target/arm/pp/kernel-pp.c` line 28 reads the
value register expressly to ack). It then runs the scheduler tick work
and returns; the timer re-fires 10 ms later on its own.

## Cache

`CACHE_CTL` at `0x6000C000`:

| Bit  | Name              | Meaning |
|------|-------------------|---------|
| 0    | `ENABLE`          | Cache on |
| 1    | `RUN`             | Cache active |
| 2    | `INIT`            | Init mode (during setup) |
| 4    | `VECT_REMAP`      | Reroute vectors to `0x6000F100` |
| 14   | `READY`           | Cache populated |
| 15   | `BUSY`            | Operation in progress |

Register addresses (`firmware/export/pp5020.h`):

| Register           | Address      | Meaning |
|--------------------|--------------|---------|
| `CACHE_PRIORITY`   | `0x60006044` | CPU/COP cache-priority bits (`0x10` CPU, `0x20` COP) |
| `CACHE_CTL`        | `0x6000C000` | Control (bits above) |
| `CACHE_MASK`       | `0xF000F040` | Cacheable-region mask |
| `CACHE_OPERATION`  | `0xF000F044` | Flush / invalidate trigger |
| `CACHE_FLUSH_MASK` | `0xF000F048` | Flush-region mask |
| `CACHE_FLUSH_BASE` | `0xF0008000` | Flush-region base |

`CACHE_OPERATION` bits:

- `0x0002` (`FLUSH`) — write back dirty lines.
- `0x0004` (`INVALIDATE`) — drop all lines.

Cache line status arrays:
- CPU cache status base: `0xF0004000`, 8 KB.
- COP cache status base: `0xF0006000`, 8 KB.

Each 32-bit status word: bits `[20:0]` line address `>> 11`, bit 22 dirty,
bit 23 valid. The cache is **unified** (one 8 KB store, 16-byte lines, holding
both instructions and data), so enabling it accelerates instruction fetch
*and* data access — but any DMA agent reading CPU-written memory then needs a
flush first (see below). No MMU; addressing is flat.

**Cache init** (`system-pp502x.c` `init_cache`), exact order:

1. `CACHE_CTL &= ~(ENABLE | RUN)`, then `CACHE_CTL |= INIT`.
2. `CACHE_PRIORITY |= 0x10` (CPU) — Rockbox does this only for non-bootloader.
3. `CACHE_MASK = 0x00001C00`; `CACHE_OPERATION = 0xFC0`.
4. `CACHE_CTL |= INIT | ENABLE | RUN`; then 4× `nop`.
5. Prime: read `CACHED_INIT_ADDR` (`0x2000`) upward for `CACHE_SIZE` (8 KB) in
   `CACHEALIGN_SIZE` (16-byte) steps — one read per line.

Init does NOT set `VECT_REMAP`, so the ARM vectors stay at `0x0`.

**DMA coherency** — the cache is write-back (each line has a dirty bit), so a
DMA engine reading a buffer the CPU just wrote (e.g. the I2S audio buffer) must
be preceded by a flush or it reads stale SDRAM. Rockbox `commit_dcache`:

```
if (CACHE_CTL & ENABLE) {
    CACHE_OPERATION |= FLUSH;         /* 0x0002 */
    while (CACHE_CTL & BUSY) { }      /* bit 15 */
    nop; nop; nop; nop;
}
```

`commit_discard_dcache` additionally invalidates (for DMA *into* cached RAM);
the audio path only writes-then-DMA-reads, so a plain flush suffices.

The PP5022 has no MMU. There's a small TLB-like remapping unit
(`MMAP0`–`MMAP7`) used for the SDRAM-to-zero remap and not much else.

## Power management

`DEV_EN` (`0x6000600C`), `DEV_EN2` (`0x60006010`), `DEV_RS`
(`0x60006004`), `DEV_RS2` (`0x60006008`), `DEV_INIT1` (`0x70000010`),
`DEV_INIT2` (`0x70000020`) gate per-peripheral clock and reset.

iPod Video boot init (`system-pp502x.c` 497–510):

```
DEV_EN    = 0xC2000124
DEV_EN2   = 0x00000000
DEV_INIT1 = 0x00000000
DEV_INIT2 = 0x40000000   ; PLL power
DEV_RS    = 0x3DFFFEF8   ; reset enabled blocks
DEV_RS2   = 0xFFFFFFFF
DEV_RS    = 0x00000000   ; release
DEV_RS2   = 0x00000000
```

**Deep sleep entry:** suspend both cores via `PROC_CTL = PROC_SLEEP`,
disable PLL, switch to 32 kHz osc. The PCF50605 PMIC handles the
final standby — see [06-power.md](06-power.md) for that side.
