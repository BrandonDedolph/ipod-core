# 01 — PortalPlayer PP5022 SoC

The iPod Video 5G/5.5G runs on the PP5022 SoC: dual ARM7TDMI cores
("CPU" and "COP" — co-processor), tightly-coupled IRAM, integrated
peripheral block, and shared external SDRAM. Despite the marketing
name, the chip behaves as a member of the PP502x family; most of the
Rockbox platform code targets PP5020/5022/5024 uniformly via
`firmware/target/arm/pp/`.

## Memory map

External SDRAM is remapped to `0x00000000` by the bootloader so
exception vectors live at low addresses. Unremapped, SDRAM lives at
`0x10000000`.

| Region        | 5G (32 MB)             | 5.5G (64 MB)           | Notes |
|---------------|------------------------|------------------------|-------|
| SDRAM         | `0x10000000`–`0x11FFFFFF` | `0x10000000`–`0x13FFFFFF` | Remapped to `0x00000000` after boot |
| ROM / flash   | `0x20000000`–`0x200FFFFF` | same                  | 1 MB; firmware partition lives below |
| MMIO          | `0x60000000`–`0x700FFFFF` | same                  | Peripherals + SoC config |
| IRAM          | `0x40000000`–`0x4001FFFF` | same                  | Tightly-coupled, ~96 KB usable to user code |
| MMAP regs     | `0xF000F000`–`0xF000F03F` | same                  | Logical→physical remap registers |

**RAM size detection** (`firmware/target/arm/pp/crt0-pp.S` lines
144–160): the bootloader writes test patterns to `0x01FFFFFF` and
`0x03FFFFFF` and reads them back to detect 32 MB vs 64 MB. The result
is preserved in `probed_ramsize` for later use (e.g., default battery
capacity differs between 5G and 5.5G).

**Memory remap (MMAP0):** `MMAP0_LOGICAL` (`0xF000F000`) is set to
`0x00000000` and `MMAP0_PHYSICAL` (`0xF000F004`) is set to `0x10000000`
with flags `0x0F84` (read/write, data/code). Done in IRAM so the
remapping code isn't pulled out from under itself.

## Clock tree

The PP5022 derives all clocks from a 24 MHz crystal through a
configurable PLL. Four canonical operating points:

| Name             | Freq      | Use                          |
|------------------|-----------|------------------------------|
| `CPUFREQ_SLEEP`  | 32 768 Hz | Deep sleep (PLL off, 32 kHz oscillator) |
| `CPUFREQ_DEFAULT`| 24 MHz    | Crystal direct, PLL bypassed |
| `CPUFREQ_NORMAL` | 30 MHz    | UI / unboosted execution     |
| `CPUFREQ_MAX`    | 80 MHz    | Boosted; codecs, decode-heavy work |

(Defined in `firmware/target/arm/pp/system-target.h`.)

**Key registers** (`firmware/export/pp5020.h`):

| Register       | Address      | Purpose |
|----------------|--------------|---------|
| `PLL_CONTROL`  | `0x60006034` | PLL enable, multiplier, post-divider |
| `PLL_STATUS`   | `0x6000603C` | Lock status (bit 31) |
| `CLOCK_SOURCE` | `0x60006020` | 4-bit nibble per clock domain (see below) |
| `DEV_INIT2`    | `0x70000020` | PLL power gate (bit 30) |
| `DEV_TIMING1`  | `0x70000034` | Memory / peripheral timing for current clock |

**`CLOCK_SOURCE` source codes** (per nibble):

| Code | Source       |
|------|--------------|
| 0    | 32 kHz       |
| 1    | 16 MHz       |
| 2    | 24 MHz crystal |
| 3    | 33 MHz       |
| 4    | 48 MHz       |
| 5    | SLOW divider |
| 6    | FAST divider |
| 7    | PLL output   |

**Frequency switch sequence** (`firmware/target/arm/pp/system-pp502x.c`
lines 379–475). Boosting from 30 MHz to 80 MHz on PP5022 — the actual
order matters because the other core sees clocks change while it's
still executing:

1. `DEV_INIT2 |= 0x40000000` — power up PLL.
2. `PLL_CONTROL |= 0x88000000` — enable PLL.
3. **Suspend the other core** via `scale_suspend_core(true)`. From this
   point until step 9 the other core is parked (see *Dual core* below).
4. `CLOCK_SOURCE = 0x20002222` — switch all domains to 24 MHz crystal
   (intermediate, no PLL dependency — keeps the bus running while we
   reconfigure the PLL).
5. `DEV_TIMING1 = 0x00000303` — relax timing for the slow clock.
6. `PLL_CONTROL = 0x8A020A03` — pre-stage write (10/3 multiplier; not
   used directly, but the controller wants this intermediate write
   before the final value).
7. `PLL_CONTROL = 0x8A121403` — program PLL for 80 MHz (final value:
   multiplier = 20/3, ÷2 post-divider, on a 24 MHz reference).
8. Spin on `PLL_STATUS & 0x80000000` until lock.
9. `CLOCK_SOURCE = 0x20007777` — route all domains to PLL output.
10. `DEV_TIMING1 = 0x00000808` — high-speed timing.
11. **Resume the other core** via `scale_suspend_core(false)`.

For 30 MHz: same shape with `PLL_CONTROL = 0x8A220501` (multiplier
5/1, ÷4 post-divider).

For sleep (32 kHz): disable post-dividers (`PLL_CONTROL |= 0x0C000000`),
switch to `CLOCK_SOURCE = 0x20000000`, disable PLL, then drop power
(`DEV_INIT2 &= ~0x40000000`).

**Boost discipline:** the firmware tracks a boost counter; UI work
runs unboosted at 30 MHz; codecs request a boost during decode. The
locking mechanism (`set_cpu_frequency__lock` /
`set_cpu_frequency__unlock`) uses a corelock to ensure the two cores
agree on the current frequency atomically.

## Dual core: CPU and COP

Both cores are ARM7TDMI; they share IRAM and SDRAM, and have a small
mailbox peripheral for IPC. Cores are distinguished by reading
`PROCESSOR_ID` at `0x60000000`:

```c
#define PROC_ID_CPU  0x55
#define PROC_ID_COP  0xAA
```

Per-core control via `CPU_CTL` (`0x60007000`) and `COP_CTL`
(`0x60007004`); the macro `PROC_CTL(core)` selects between them.

**Sleep / wake:**

| Bit          | Value         | Meaning |
|--------------|---------------|---------|
| `PROC_SLEEP` | `0x80000000`  | Sleep until interrupt |
| `PROC_WAIT_CNT` | `0x40000000` | Sleep until countdown |
| `PROC_WAKE_INT` | `0x20000000` | Fire IRQ on wake |
| `PROC_CNT_USEC` | `0x02000000` | Counter ticks at 1 µs |
| `PROC_CNT_MSEC` | `0x01000000` | Counter ticks at 1 ms |
| `[7:0]`      | —             | Read: cycles remaining; write: cycles to skip |

**Always insert 3 NOPs after writing PROC_CTL** — the pipeline needs
a beat for the write to take effect.

**Mailbox IPC** (registers at `0x60001000`):

| Register       | Address      | Behavior |
|----------------|--------------|----------|
| `MBX_MSG_STAT` | `0x60001000` | Read mailbox bits |
| `MBX_MSG_SET`  | `0x60001004` | Atomic OR (set bits) |
| `MBX_MSG_CLR`  | `0x60001008` | Atomic AND-NOT (clear bits) |
| `CPU_QUEUE`    | `0x60001010` | CPU writes bit 29 to signal COP |
| `COP_QUEUE`    | `0x60001020` | COP writes bit 29 to signal CPU |

Wake protocol (`firmware/target/arm/pp/thread-pp.c` lines 319–346):

1. Set wake-intent flags via `MBX_MSG_SET`.
2. Spin until target core signals sleep-intent (`0x4 << core_id`).
3. If target's `PROC_CTL` shows `PROC_SLEEP`, write 0 to clear it.
4. Clear wake procedure flag via `MBX_MSG_CLR`.

**COP boot:** the bootloader (running on CPU) wakes the COP after
SDRAM remap (`crt0-pp.S` lines 193–214). The COP redoes the remap on
its side and sets up its own stacks. CPU then waits for the COP to
finish kernel init before either core can do meaningful work.

**Core suspension during clock change** (`scale_suspend_core`,
`system-pp502x.c` 330–348): the active core writes `0x40000000` to the
*other* core's `PROC_CTL` (signal-wait), then writes `0x48000003` to
its own (stall 3 cycles); on resume, the active core stalls 31 cycles
and clears the other's control word. Interrupts are disabled across
the whole sequence.

## Interrupt controller

Two priority levels (low and high) per core, with separate enable /
disable / status registers.

**Low-priority** (`firmware/export/pp5020.h` lines 66–137):

| Register          | Address      | Purpose |
|-------------------|--------------|---------|
| `CPU_INT_STAT`    | `0x60004000` | CPU IRQ status |
| `COP_INT_STAT`    | `0x60004004` | COP IRQ status |
| `CPU_FIQ_STAT`    | `0x60004008` | CPU FIQ status |
| `COP_FIQ_STAT`    | `0x6000400C` | COP FIQ status |
| `INT_STAT`        | `0x60004010` | Shared status (logical OR) |
| `CPU_INT_EN`      | `0x60004024` | Write 1 to enable a bit |
| `CPU_INT_DIS`     | `0x60004028` | Write 1 to disable a bit |
| `CPU_INT_PRIORITY`| `0x6000402C` | Per-IRQ priority |
| `COP_INT_EN`      | `0x60004034` | (analogous for COP) |
| `COP_INT_DIS`     | `0x60004038` | |

**High-priority** (offset `+0x100`):

| Register         | Address      |
|------------------|--------------|
| `CPU_HI_INT_STAT`| `0x60004100` |
| `CPU_HI_INT_EN`  | `0x60004124` |
| `CPU_HI_INT_DIS` | `0x60004128` |

**IRQ numbers** of interest. The IRQ number is the absolute index;
sources at `# < 32` route through `CPU_INT_EN` / `CPU_INT_DIS` directly
(bit `# = N`); sources at `# ≥ 32` route through `CPU_HI_INT_EN` /
`CPU_HI_INT_DIS` at bit `(N − 32)`.

| #  | Source                  | Bank | Register bit |
|----|-------------------------|------|---------------|
| 0  | `TIMER1_IRQ`            | low  | `1 << 0`      |
| 1  | `TIMER2_IRQ`            | low  | `1 << 1`      |
| 4  | `MAILBOX_IRQ`           | low  | `1 << 4`      |
| 10 | `IIS_IRQ` (I²S audio)   | low  | `1 << 10`     |
| 20 | `USB_IRQ`               | low  | `1 << 20`     |
| 23 | `IDE_IRQ` (ATA)         | low  | `1 << 23`     |
| 25 | `FIREWIRE_IRQ`          | low  | `1 << 25`     |
| 26 | `DMA_IRQ`               | low  | `1 << 26`     |
| 30 | `HI_IRQ` (chained high) | low  | `1 << 30`     |
| 32 | `GPIO0_IRQ` (ports A–D) | high | `1 << 0`      |
| 33 | `GPIO1_IRQ` (ports E–H) | high | `1 << 1`      |
| 34 | `GPIO2_IRQ` (ports I–L) | high | `1 << 2`      |
| 36 | `SER0_IRQ` (UART 0)     | high | `1 << 4`      |
| 40 | `I2C_IRQ`               | high | `1 << 8`      |

So enabling `I2C_IRQ` from C looks like:

```c
CPU_HI_INT_EN = (1u << (40 - 32));   // == 1 << 8 == 0x100
```

This indirection is easy to get wrong; Rockbox's source uses
`#define I2C_MASK (1 << (I2C_IRQ - 32))` and similar macros to keep
each driver from re-deriving it.

**Vector remap:** `CACHE_CTL` bit `0x10` (`CACHE_CTL_VECT_REMAP`)
diverts the standard ARM exception vectors to `0x6000F100`; entries
are `INT_VECTOR_ENTRY(v)` = `*((vu32*)0x6000F100 + v)`.

## Cache

`CACHE_CTL` at `0x6000C000`:

| Bit  | Name              | Meaning |
|------|-------------------|---------|
| 0    | `ENABLE`          | Cache on |
| 1    | `RUN`             | Cache active |
| 2    | `INIT`            | Init mode (during setup) |
| 4    | `VECT_REMAP`      | Reroute vectors to `0x6000F100` |
| 14   | `READY`           | Cache populated |
| 15   | `BUSY`            | Operation in progress |

`CACHE_OPERATION` at `0xF000F044` accepts:

- `0x0002` (`FLUSH`) — write back dirty lines.
- `0x0004` (`INVALIDATE`) — drop all lines.

Cache line status arrays:
- CPU cache status base: `0xF0004000`, 8 KB.
- COP cache status base: `0xF0006000`, 8 KB.

Each 32-bit status word: bits `[20:0]` line address `>> 11`, bit 22 dirty,
bit 23 valid.

**Cache init** (`system-pp502x.c` lines 289–323):

1. Clear `ENABLE` and `RUN`, set `INIT`.
2. `CACHE_MASK = 0x00001C00`, `CACHE_OPERATION = 0xFC0`.
3. `CACHE_CTL |= INIT | ENABLE | RUN`.
4. Prime cache by reading `0x2000` upward in 16-byte chunks (8 KB total).

The PP5022 has no MMU. There's a small TLB-like remapping unit
(`MMAP0`–`MMAP7`) used for the SDRAM-to-zero remap and not much else.

## Power management

`DEV_EN` (`0x6000600C`), `DEV_EN2` (`0x60006010`), `DEV_RS`
(`0x60006004`), `DEV_RS2` (`0x60006008`), `DEV_INIT1` (`0x70000010`),
`DEV_INIT2` (`0x70000020`) gate per-peripheral clock and reset.

iPod Video boot init (`system-pp502x.c` 497–510):

```
DEV_EN    = 0xC2000124
DEV_EN2   = 0x00000000
DEV_INIT1 = 0x00000000
DEV_INIT2 = 0x40000000   ; PLL power
DEV_RS    = 0x3DFFFEF8   ; reset enabled blocks
DEV_RS2   = 0xFFFFFFFF
DEV_RS    = 0x00000000   ; release
DEV_RS2   = 0x00000000
```

**Deep sleep entry:** suspend both cores via `PROC_CTL = PROC_SLEEP`,
disable PLL, switch to 32 kHz osc. The PCF50605 PMIC handles the
final standby — see [06-power.md](06-power.md) for that side.
