# 09 — On-SoC I²C controller (PP502x)

The PP5022 has a small dedicated I²C master block. On the iPod Video its
only Phase-1 consumer is the **WM8758 audio codec** (control interface;
see [05-audio.md](05-audio.md)), but the block is general-purpose — the
PMIC and accelerometer sit on the same bus on other targets — so it gets
its own subsystem doc.

The controller is a fixed-shape "load address, load up to four data
bytes, set count, strobe" engine. It does **not** expose SCL/SDA
bit-banging; you hand it a device address and a payload and it clocks the
transfer out. There is no receive-side complexity we need for the codec
(the WM8758 control port is write-only in our use).

> **Not the clickwheel.** The touch-wheel "OPTO" block lives one page over
> at `0x7000C100`–`0x7000C140` and is a *different* peripheral with a
> different layout and a different clock gate — see
> [03-clickwheel.md](03-clickwheel.md). Do not conflate the two. This doc
> is exclusively the I²C controller at `0x7000C000`.

## Register block

Base `I2C_BASE = 0x7000C000`. **All controller registers are byte-wide**
(8-bit accesses); only the clock/reset gating (below) is 32-bit.

| Register     | Address      | Purpose |
|--------------|--------------|---------|
| `I2C_CTRL`   | `0x7000C000` | Start strobe (bit 7), read/write select (bit 5), byte-count field (bits 2:1) |
| `I2C_ADDR`   | `0x7000C004` | 7-bit device address in bits 7:1; bit 0 = R/W (0 = write) |
| `I2C_DATA0`  | `0x7000C00C` | Payload byte 0 |
| `I2C_DATA1`  | `0x7000C010` | Payload byte 1 |
| `I2C_DATA2`  | `0x7000C014` | Payload byte 2 |
| `I2C_DATA3`  | `0x7000C018` | Payload byte 3 |
| `I2C_STATUS` | `0x7000C01C` | Bit 6 = `BUSY` |

Data slots are at `I2C_BASE + 0x0C + 4*n` (stride 4). Only four exist:
**a single transaction carries at most 4 payload bytes.** Larger transfers
must be chunked by the caller.

Source: `firmware/target/arm/pp/i2c-pp.c:40-43` (register macros),
`firmware/export/pp5020.h:560` (`I2C_BASE`).

### Control / status bit values

| Symbol       | Register     | Value  | Meaning |
|--------------|--------------|--------|---------|
| `I2C_SEND`   | `I2C_CTRL`   | `0x80` | Bit 7: strobe — begin the queued transaction |
| `I2C_READ`   | `I2C_CTRL`   | `0x20` | Bit 5: 1 = read, 0 = write |
| `I2C_COUNT`  | `I2C_CTRL`   | `0x06` | Bits 2:1: `(len − 1)`, i.e. bytes-minus-one |
| `I2C_RW`     | `I2C_ADDR`   | `0x01` | Bit 0 of the address byte: 1 = read, 0 = write |
| `I2C_BUSY`   | `I2C_STATUS` | `0x40` | Bit 6: 1 = transfer in progress, 0 = idle/ready |

Source: `i2c-pp.c:46` (`I2C_SEND`), `:49` (`I2C_BUSY`), `:90,:137`
(read-select bit 5), `:92,:144` (count field), `:88,:135` (address R/W bit).

The read-select bit (bit 5) carries no symbolic name in the reference; it
is inferred with high confidence from the read-vs-write asymmetry (set on
reads, cleared on writes). The other CTRL bits are as named.

## Write transaction

The controller frames "how many bytes" itself via the CTRL count field;
that is a layer *below* any device-specific register framing (the WM8758's
own 9-bit control word is a codec convention layered on top — see
[05-audio.md](05-audio.md), "DAC I²C framing").

To write `len` bytes (1 ≤ `len` ≤ 4) to 7-bit device address `dev`:

1. Wait until `I2C_STATUS & I2C_BUSY` is clear (bounded — see timeout).
2. Enter a critical section (IRQs off) so a preemption can't interleave a
   second transaction into the shared register file.
3. `I2C_ADDR = (dev & 0x7F) << 1` — device in bits 7:1, R/W bit 0 clear
   (write).
4. Clear the read-select bit: `I2C_CTRL &= ~I2C_READ`.
5. Store each payload byte: `I2C_DATA(n) = data[n]` for `n` in `0..len-1`.
6. Set the count field: `I2C_CTRL = (I2C_CTRL & ~I2C_COUNT) | ((len-1) << 1)`.
7. Strobe: `I2C_CTRL |= I2C_SEND`.
8. Leave the critical section.

The write path does **not** wait for completion after the strobe — the
*next* transaction's leading BUSY-wait (step 1) covers it. This is fine for
a burst of codec register writes, but if a write is the last bus activity
before you touch the codec through some other channel, poll BUSY yourself
afterward.

Source: `i2c-pp.c:117-152` (`pp_i2c_send_bytes`, write direction).

### Read transaction (recorded for completeness)

Reads set the R/W bit in `I2C_ADDR` (`| 0x01`) and the read-select bit in
`I2C_CTRL` (`| I2C_READ`), strobe, then **wait for BUSY to clear before
reading back** `I2C_DATA(0..len-1)`. The codec control port is write-only
in our use, so the read path is not needed for Phase 1.

Source: `i2c-pp.c:70-115` (`pp_i2c_read_bytes`).

### BUSY poll / timeout

Poll `I2C_STATUS`; idle when `(I2C_STATUS & I2C_BUSY) == 0`. Rockbox bounds
the wait at ~1 second (`HZ` ticks) and fails the transfer on timeout. Our
freestanding driver has no scheduler tick to count in the bring-up path, so
it uses a bounded spin-count in the same spirit as the UART TX and PLL-lock
spins (a dead bus must not hang the core).

Source: `i2c-pp.c:49` (`I2C_BUSY`), `:56-68` (`pp_i2c_wait_not_busy`,
`POLL_TIMEOUT = HZ`).

## Controller init

One-time bring-up before any transfer. The I²C block is gated purely from
the device enable/reset registers in the `0x60006000` clock block — **not**
from `DEV_INIT1/2`.

1. Clock-gate on: `DEV_EN |= DEV_I2C` (`DEV_I2C = 0x00001000`, bit 12) at
   `DEV_EN = 0x6000600C`.
2. Reset pulse: `DEV_RS |= DEV_I2C` then `DEV_RS &= ~DEV_I2C` at
   `DEV_RS = 0x60006004` — assert then release the block reset.
3. Clock/config poke: write `0x00000000` then `0x00000080` to the
   undocumented I²C clock register at `0x600060A4`. This has no symbolic
   name in the reference (it sits adjacent to `CLCD_CLOCK_SRC` at
   `0x600060A0`); `0x80` is the value the iPod path uses. Treat it as
   required magic.
4. A throwaway warm-up read (Rockbox reads one byte from device `0x08`) to
   prime the controller before real traffic.

Source: `i2c-pp.c:250-315` (`i2c_init`, iPod `I2C_PP5020` path),
`pp5020.h:159` (`DEV_I2C`), `:148,:150` (`DEV_RS`/`DEV_EN`).

## Device-family note

The iPod 5.5G is a **PP5022** but selects the PP5020 I²C code path
(`CONFIG_I2C = I2C_PP5020`, `config/ipodvideo.h:181`), so every address and
bit above is the PP5020 value and applies verbatim. **The PP5002 I²C block
is different** (`I2C_BASE = 0xC0008000`, different enable register/bit) and
none of its addresses apply here.
